## TODO 2: Paystack fund flow verification (PRE-PHASE-2 — HIGH PRIORITY)

**What:** Contact Paystack support and confirm: (1) does their split settlement API support deducting a platform fee at transaction time? (2) can the remainder go directly to the estate's registered bank account in a single transaction?

**Why:** If Paystack does not support split settlement, the platform needs an escrow wallet (middleware that holds funds, then disburses). This is a significant additional sprint and changes the regulatory exposure (holding community funds may require a stored-value license in Nigeria).

**Pros:** Locks in Phase 2 architecture before it starts. Prevents mid-build redesign.

**Cons:** Paystack support response time may be slow. May discover the answer requires legal review.

**Context:** The design doc lists fund flow as "Open Question #2" but treats it as a minor detail. The outside voice correctly identified it as a blocking architectural decision for Phase 2.

**How to apply:** Email Paystack merchant support with the specific question. Check Paystack's developer documentation for "subaccounts" and "split payment" — this feature exists but verify the fee-deduction direction (platform takes fee before disbursement).

**Blocked by:** Nothing. **Blocking:** Phase 2 payment integration design.

## TODO 3: Vercel Lagos latency test (PRE-DEPLOY — MEDIUM PRIORITY)

**What:** Run a latency test from Lagos to Vercel before the first production deployment. If p95 latency > 400ms, evaluate Railway with a closer server region.

**Why:** Nigerian users on mobile data networks will abandon slow apps. The public summary link in particular gets high traffic (WhatsApp shares) and must load fast on slow connections.

**Pros:** 30 minutes. Prevents a deployment migration later.

**Cons:** None significant.

**Context:** Vercel's primary servers are US-based. Vercel has edge nodes globally but Next.js server-side rendering still hits origin for dynamic routes. A latency test from Lagos reveals actual performance before the app is in production.

**How to apply:** Ask someone with a Nigerian IP (or use a Nigerian VPN endpoint) to run:
`curl -o /dev/null -s -w '%{time_total}\n' https://[your-vercel-app].vercel.app`
Run 10 times. If median > 400ms, evaluate Railway (has Frankfurt and Singapore regions).

**Blocked by:** Nothing. **Blocking:** Final deployment platform decision.

## TODO 4: Levy detail page design (PRE-PHASE-1-IMPLEMENTATION — HIGH PRIORITY)

**What:** Design and specify the levy detail screen — what the chairman sees when they tap into a specific levy from the dashboard. Must show: all residents, their payment status (paid/unpaid/partial/overdue/disputed), ability to mark individual payments received, and the "pending reviews" queue for uploaded bank transfers.

**Why:** The "mark payment received" flow is core to the manual payment recording feature in Phase 1. Without specifying this screen, the engineer will invent it — and probably get it wrong. The levy detail page is also where the chairman spends most of their time during a levy cycle.

**Pros:** Removes the biggest ambiguous screen in Phase 1 before implementation starts.

**Cons:** 1-2 hours of design work (another wireframe pass or a detailed spec).

**Context:** Identified in /plan-design-review Pass 7. Chairman dashboard links to levy detail but the screen is unspecified. Contains: resident list + status badges + "Review transfer" action for partial rows + "Mark paid" quick action.

**Blocked by:** Nothing. **Blocking:** Phase 1 levy management implementation.
